## Context

商品 preprocess 主入口 `data_preprocess/script_preprocess/future_upgraded/commodity/main.sh` 当前将整条 `run_commodity_full_process` 输出重定向到一个总日志文件。实际主流程由 `fu_full_process.sh` 串联多个长耗时步骤，部分步骤内部又产生按日期的子日志。总日志可以记录整体输出，但不能稳定提供每个主要阶段的独立日志入口，长流程失败时定位成本高。

## Goals / Non-Goals

**Goals:**
- 为商品 preprocess 的主要阶段生成独立步骤日志文件。
- 在总日志中记录每个步骤的开始、成功、失败和步骤日志路径。
- 保持现有 `main.sh` 使用方式、产物路径和失败停止语义。
- 保留已有 cross-section/merge 按日期子日志，不替换它们。

**Non-Goals:**
- 不修改 Python 预处理算法、数据格式或输出目录。
- 不引入新日志框架、外部依赖、日志轮转或压缩策略。
- 不改变 Binance/crypto futures 预处理脚本。

## Decisions

1. 在 shell 层增加统一步骤包装函数，而不是在每个 Python 脚本里散落日志逻辑。
   - 理由：本次需求是主流程阶段可观测性，shell 层可以覆盖 Python 模块和已有 shell 函数，改动集中且不影响数据逻辑。
   - 备选：逐个 Python 模块加 logging。该方案覆盖面更细，但会扩大实现范围并改变多个模块的日志行为。

2. 每个主要阶段输出独立日志文件，同时总日志记录阶段状态。
   - 理由：独立日志便于直接定位失败阶段，总日志保留整体调度视图。
   - 备选：只打印总日志阶段 begin/end。该方案简单，但仍需要在总日志中搜索大量输出。

3. 步骤日志文件名包含 `symbol`、`target_freq`、`start_date`、`end_date` 和步骤名。
   - 理由：同一目录中可以区分不同运行参数和阶段，避免日志含义不清。
   - 备选：每次运行新建 timestamp 目录。该方案更隔离，但会引入日志清理和查找复杂度，本次不需要。

4. 步骤失败时返回原始失败状态并停止主流程。
   - 理由：商品 preprocess 当前依赖 fail-fast 暴露数据或流程问题，日志增强不能掩盖失败。
   - 备选：失败后继续执行后续步骤。该方案会产生下游连锁错误，不适合预处理流水线。

## Risks / Trade-offs

- 步骤日志和总日志会产生重复的阶段状态信息 -> 接受该重复，换取排障入口清晰。
- cross-section/merge 已有按日期子日志，新增阶段日志可能只是汇总 nohup 调度输出 -> 总日志和阶段日志仍能告诉用户该阶段是否启动、跳过或等待完成，日期级细节继续在已有子日志中查看。
- 如果步骤日志文件名固定，重复运行相同参数会覆盖上次日志 -> 符合现有总日志按参数覆盖的模式；本次不引入日志归档策略。

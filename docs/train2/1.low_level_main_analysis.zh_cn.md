# script/test/DiHFT/low_level/main.sh 逻辑流程说明

本文档说明 `script/test/DiHFT/low_level/main.sh` 及其调用链的作用。该脚本用于回测阶段 I 训练出的低层 ensemble 模型，并为后续“过滤集成模型”生成 `analysis_result.npy`。

在 FineFT 流程中，它处在阶段 II 的第一步：

```text
阶段 I 训练低层 ensemble agent
        |
        v
阶段 II 回测 ensemble 中每个 context/bin 的验证集表现
        |
        v
过滤脚本根据回测结果重组 potential model
        |
        v
阶段 III 高层 VAE/启发式路由选择低层 agent
```

## 1. 脚本定位

入口脚本：

```text
script/test/DiHFT/low_level/main.sh
```

它本身只做一件事：后台启动 `test_util.sh`，并把日志写到文件中。

```bash
nohup bash script/test/DiHFT/low_level/test_util.sh \
    >log/DiHFT/trading_pair/low_level/test_pes_rest.log 2>&1 &
```

真正的回测逻辑分布在：

- `script/test/DiHFT/low_level/test_util.sh`：为不同交易对和 epoch 批量启动回测任务。
- `RL/DiHFT/low_level/test_agent_index.py`：逐个验证集动态、初始动作和 context/bin 回测 ensemble 子策略。
- `model/low_level.py`：定义 `ensemble_Qnet`。
- `env/env_initiate/base_initiate.py` 与 `env/env_class/futures_util.py`：构造交易环境和动作/持仓映射。

## 2. 输入与输出

### 2.1 输入

默认读取的数据和模型包括：

```text
dataset/<dataset_name>/valid/<label>/df_*.feather
dataset/<dataset_name>/state_features.npy
dataset/<dataset_name>/maintenance_margin_ratio_dict.npy

result/DiHFT/low_level/<dataset_name>/weights_advantage_pretrain/epoch_<k>/trained_model.pkl
```

其中：

- `valid/<label>/df_*.feather`：按市场动态标签切分后的验证集片段。
- `state_features.npy`：交易环境使用的状态特征列名。
- `maintenance_margin_ratio_dict.npy`：维持保证金规则。
- `trained_model.pkl`：阶段 I 保存的低层 ensemble Q 网络参数。

### 2.2 输出

每个被回测的 epoch 会在对应模型目录下保存：

```text
result/DiHFT/low_level/<dataset_name>/weights_advantage_pretrain/epoch_<k>/analysis_result.npy
```

`analysis_result.npy` 是后续过滤脚本的核心输入。它保存的是一个 list，每个元素对应一组：

```text
市场动态 label
x 初始动作 initial_action
x ensemble context/bin_index
```

记录结构如下：

```python
{
    "label": label,
    "initial_action": initial_action,
    "bin_index": bin_index,
    "reward_sum": [...],
    "df_length": [...],
    "turnover": [...],
}
```

字段含义：

- `label`：验证集动态标签，例如 `label_0`。
- `initial_action`：测试开始时的初始动作，对应初始持仓。
- `bin_index`：ensemble 中第几个 context/Q 子网络。
- `reward_sum`：该 context 在多个验证片段上的累计 reward。
- `df_length`：每个验证片段长度。
- `turnover`：动作切换幅度统计，用于衡量换手。

## 3. main.sh 执行逻辑

`main.sh` 使用 `nohup` 后台运行：

```text
script/test/DiHFT/low_level/test_util.sh
```

日志重定向到：

```text
log/DiHFT/trading_pair/low_level/test_pes_rest.log
```

注意这里的日志目录需要提前存在。如果没有创建 `log/DiHFT/trading_pair/low_level/`，shell 重定向会失败，后台任务不会正常启动。

## 4. test_util.sh 批量调度逻辑

`test_util.sh` 定义了两个函数：

```bash
run_ddqn_context <dataset_name> <max_holding_number> <epoch_start> <epoch_end>
run_ddqn_average <dataset_name> <max_holding_number> <epoch_start> <epoch_end>
```

当前实际启用的是 `run_ddqn_context`：

```bash
run_ddqn_context DOTUSDT 6000 1 50
run_ddqn_context ETHUSDT 160 1 50
```

其他交易对命令被注释掉，可以按需要打开：

```text
BTCUSDT: max_holding_number = 8
ETHUSDT: max_holding_number = 160
BNBUSDT: max_holding_number = 100
DOTUSDT: max_holding_number = 6000
```

### 4.1 run_ddqn_context

该函数会遍历指定 epoch 范围：

```bash
for epoch in $(seq $epoch_start $epoch_end); do
    CUDA_VISIBLE_DEVICES=${gpu_index} nohup python RL/DiHFT/low_level/test_agent_index.py \
        --dataset_name "${dataset_name}" \
        --max_holding_number "${max_holding_number}" \
        --epoch_num "${epoch}" \
        >"${log_dir}/epoch_${epoch}.log" 2>&1 &
done
```

GPU 分配规则：

```bash
gpu_index=$(((epoch - 1) % 4))
```

也就是 epoch 轮流分配到 GPU 0、1、2、3。

日志路径：

```text
log/DiHFT/<dataset_name>/low_level/test/epoch_<k>.log
```

注意：`run_ddqn_context` 中等待进程完成的 `wait` 逻辑被注释掉了。因此函数会批量提交后台任务后直接返回，不会阻塞等待所有 epoch 完成。

### 4.2 run_ddqn_average

`run_ddqn_average` 会调用：

```text
RL/DiHFT/low_level/test_agent_average.py
```

它当前没有在脚本末尾启用。主线阶段 II 使用的是 `test_agent_index.py`，因为后续过滤脚本需要按 `bin_index` 比较每个 context 的表现。

## 5. test_agent_index.py 初始化流程

核心类：

```python
class weighted_trader
```

初始化时会完成以下工作：

1. 根据 CUDA 是否可用选择设备。
2. 定位模型路径：

```text
result/DiHFT/low_level/<dataset_name>/weights_advantage_pretrain/epoch_<epoch_num>/
```

3. 读取验证数据路径：

```text
dataset/<dataset_name>/valid/
```

4. 读取状态特征和维持保证金参数。
5. 根据 `max_holding_number` 构造持仓档位。
6. 构造 `ensemble_Qnet` 并加载 `trained_model.pkl`。

默认动作空间：

```text
position_choices = 9
leverage_choices = [5]
N_ACTIONS = 9
```

默认 context 数量：

```text
N = 7
```

因此模型输出形状为：

```text
batch_size x 7 x 9
```

`bin_index` 表示选择第几个 context 的 Q 值来决策。

## 6. 持仓与初始动作

代码会根据 `max_holding_number` 和 `position_choices=9` 构造 9 个目标持仓档位。

例如 `BTCUSDT` 的 `max_holding_number=8` 时，持仓档位大致为：

```text
[-8, -6, -4, -2, 0, 2, 4, 6, 8]
```

回测会遍历：

```python
self.initial_action_list = range(9)
```

也就是每个模型不仅要在不同市场动态上测试，还要在不同初始持仓条件下测试。

## 7. 单步决策逻辑

函数：

```python
act_test(state, info, context_index)
```

输入包括：

- 当前市场状态 `state`
- 上一步动作 `previous_action`
- 当前可用动作 mask `avaliable_action`
- funding 倒计时小时和分钟
- 当前使用的 `context_index`

执行流程：

```text
state/info
   |
   v
ensemble_Qnet 输出所有 context 的 Q 值
   |
   v
取 actions_value[:, context_index, :]
   |
   v
argmax 得到当前动作
```

可用动作 mask 已经在 `Qnet.forward()` 中处理：

```python
masked_action = action + (avaliable_action - 1) * MAX_PUNISHMENT
```

不可用动作会被加上极大负惩罚，从而不会被 `argmax` 选中。

## 8. 回测主循环

入口：

```python
trader = weighted_trader(args)
trader.test()
```

`test()` 的嵌套循环可以概括为：

```text
for label in valid/:
    for initial_action in 0..8:
        for bin_index in 0..N-1:
            for df_path in valid/<label>/:
                构造交易环境
                从 initial_action 映射初始持仓和杠杆
                从头到尾运行该验证片段
                记录 reward_sum、df_length、turnover
```

其中每个验证片段会：

1. 读取 `df_*.feather`。
2. 根据初始动作映射初始持仓和杠杆。
3. 根据当前价格计算初始保证金：

```python
initial_margin = abs(initial_position * current_markprice / initial_leverage)
```

4. 调用 `initiate_base_env()` 创建环境。
5. 循环执行 `act_test()` 和 `env.step()`，直到片段结束。

## 9. turnover 统计

每一步都会累加：

```python
turn_over += abs(a - previous_action) / 4
```

这里的 `a` 和 `previous_action` 是动作编号。除以 4 是因为默认 9 个动作对应从最大空头到最大多头，中间相邻档位代表一定持仓间隔；该值是一个近似换手指标。

最终每个验证片段保存一个 `turn_over` 值。

## 10. 保存 analysis_result.npy

每组 `label + initial_action + bin_index` 在遍历完该 label 下所有验证片段后，会追加一条记录到 `overall_result`：

```python
overall_result.append(
    {
        "label": label,
        "initial_action": initial_action,
        "bin_index": bin_index,
        "reward_sum": reward_sum_list,
        "df_length": df_length_list,
        "turnover": turnover_list,
    }
)
```

最后保存：

```python
np.save(os.path.join(self.epoch_path, "analysis_result.npy"), overall_result)
```

因此每个 epoch 会有一个独立的回测结果文件。

## 11. 完整流程图

```text
main.sh
  |
  v
nohup test_util.sh
  |
  v
run_ddqn_context(dataset, max_holding_number, epoch_start, epoch_end)
  |
  v
为每个 epoch 后台启动 test_agent_index.py
  |
  v
加载 epoch_<k>/trained_model.pkl
  |
  v
遍历 valid/<label>/df_*.feather
  |
  v
遍历 initial_action 和 bin_index
  |
  v
在交易环境中逐步选择动作并回测
  |
  v
保存 epoch_<k>/analysis_result.npy
```

## 12. 需要注意的实现细节

1. `main.sh` 的顶层日志目录如果不存在，重定向会失败。可以先创建：

```bash
mkdir -p log/DiHFT/trading_pair/low_level
```

2. `run_ddqn_context` 没有等待后台进程结束。运行过滤脚本前，需要确认所有目标 epoch 的 `analysis_result.npy` 都已经生成。

3. 当前 `test_util.sh` 实际只启用了 `DOTUSDT` 和 `ETHUSDT`。如需回测 `BTCUSDT` 或 `BNBUSDT`，需要取消对应命令注释。

4. `test_agent_index.py` 默认读取 `weights_advantage_pretrain` 目录。如果要回测其他低层模型目录，需要同步修改模型路径逻辑。

5. `position_choices`、`leverage_choices`、`hidden_nodes`、`N` 等参数需要和训练阶段保持一致，否则加载模型或解释动作空间时会不匹配。
